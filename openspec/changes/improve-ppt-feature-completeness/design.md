## Context

当前 PPT 能力横跨多个模块：`app/(main)/ppt` 页面编排、`components/ppt` 交互组件、`hooks/ppt` 生成流程、`stores/tools/ppt-editor-store` 编辑状态、`stores/workflow` 演示稿生命周期、`lib/ai/workflows` 生成执行与 `lib/export/document/pptx-export` 导出。现状问题不是单点 bug，而是跨模块行为在不同入口和边界状态下出现能力缺口，导致“有实现但不完整”。

本次设计目标是以“功能完整性”为核心，把 PPT 主链路拆解为可核查的能力单元，先建立统一基线，再逐模块补齐并用自动化测试锁定行为。

## Goals / Non-Goals

**Goals:**
- 为 PPT 全流程建立统一完整性基线（创建、编辑、预览、播放、导出、管理、错误恢复）。
- 保证同一能力在不同入口和模式下行为一致（首页卡片、编辑器、预览弹窗、播放模式等）。
- 消除静默降级：任何失败分支都必须有明确反馈和可恢复动作。
- 建立“场景 -> 规格 -> 测试”一一对应关系，防止后续再出现简化实现。

**Non-Goals:**
- 不重写 PPT 编辑器架构，不替换现有 Zustand/Next.js/Tauri 技术栈。
- 不新增与“完整性补齐”无关的大型功能（例如全新模板引擎或全新渲染内核）。
- 不对非 PPT 功能域做 UI 风格重构。

## Decisions

### Decision 1: 建立“模块级完整性清单 + 场景矩阵”作为唯一验收源
- 方案：按 `Entry/Creation/Editor/Preview/Slideshow/Export/Presentation Management` 七个域建立检查矩阵，每个域定义“必须存在的能力、异常路径、验证方式”。
- 原因：用户诉求是“逐个部分检查且无省略”，需要结构化核查而非零散修补。
- 备选：按文件目录逐个修复缺陷。
- 未选原因：无法证明“完整”，也容易遗漏跨入口行为。

### Decision 2: 统一状态语义，页面与组件仅消费规范化状态
- 方案：将 `workflow-store` 与 `ppt-editor-store` 的核心状态（active presentation、dirty state、generation/export status、error state）做规范化映射，入口页面、预览、播放等均从同一语义层读取。
- 原因：当前缺口主要来自多入口分支对状态解释不一致。
- 备选：各入口各自处理状态。
- 未选原因：会继续产生行为漂移，增加后续维护成本。

### Decision 3: 显式错误面与恢复动作成为强制要求
- 方案：对生成、导出、保存、加载、删除等关键操作统一定义错误反馈和恢复动作（重试、回退、提示下一步）。
- 原因：静默失败会被用户视为“功能被省略”。
- 备选：仅日志记录错误。
- 未选原因：不满足完整可用与可诊断要求。

### Decision 4: 规格驱动测试覆盖，优先端到端闭环
- 方案：每个规格场景至少落地一个测试；跨模块关键链路优先 E2E（创建 -> 编辑 -> 预览/播放 -> 导出），模块内部行为用 Jest 补充。
- 原因：完整性问题通常出现在模块边界，单测不足以证明全链路可用。
- 备选：仅新增单测。
- 未选原因：无法覆盖真实用户路径中的集成问题。

## Risks / Trade-offs

- [Risk] 状态统一过程中可能改变现有 UI 的边界行为。
  - Mitigation: 采用增量迁移；先建立兼容映射与回归测试，再清理旧分支。
- [Risk] 补齐错误反馈后，短期内暴露更多历史问题。
  - Mitigation: 对错误分级展示，先确保可恢复，再逐步优化错误文案与可观测性。
- [Trade-off] E2E 用例增加会拉长 CI 时间。
  - Mitigation: 保留关键闭环场景为 E2E，其余逻辑下沉到 Jest 进行快速验证。

## Migration Plan

1. 产出完整性矩阵与缺口清单，标记每个模块“已完整/缺失/简化”。
2. 优先补齐跨入口状态语义与错误反馈（高影响、低业务入侵）。
3. 按链路顺序补齐功能：Creation -> Editor -> Preview/Slideshow -> Export -> Management。
4. 同步补充测试并建立规格映射关系，确保每个必须能力均可自动验证。
5. 执行一轮端到端回归，确认 apply 前所有 required artifacts 对应行为可落地。

## Open Questions

- PPT 导出失败是否需要统一错误码（便于 UI 展示可操作建议）？
- 预览模式与编辑模式间是否需要更严格的状态冻结策略，避免并发修改导致差异？
- 是否需要在后续引入“完整性健康检查脚本”自动扫描关键 capability 的测试覆盖与入口可达性？
