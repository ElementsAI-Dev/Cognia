## Context

现有工作流相关能力分散在 `app/workflows`、`components/workflow`、`stores/workflow`、`hooks/workflow` 与调度模块中。当前问题不是单点缺陷，而是端到端链路在不同入口和状态下存在不完整实现：某些操作路径缺失、某些状态不一致、某些错误分支仅部分处理，导致用户感知为“功能被省略或简化”。

本次变更以“完整性优先”为原则，先定义统一功能基线，再按基线逐段补齐实现与测试，覆盖 Web 前端状态层与（必要时）Tauri 运行时反馈映射。

## Goals / Non-Goals

**Goals:**
- 定义工作流全链路的最小完整功能集，并作为统一验收标准。
- 保证创建、编辑、执行、暂停/恢复、取消、结果查看、历史追踪、调度与触发同步在所有入口中行为一致。
- 消除静默降级：任何不支持/失败的路径都必须有明确反馈与可恢复动作。
- 用自动化测试（单测+E2E）锁定关键场景，防止回归。

**Non-Goals:**
- 不重写整个工作流编辑器或运行时架构。
- 不引入新的工作流 DSL 或新的执行引擎。
- 不对无关业务模块做 UI 重设计。

## Decisions

### Decision 1: 先建立“完整性清单”，再逐模块补齐
- 方案：把工作流能力拆成生命周期阶段（Authoring、Execution、Observability、Scheduling），每阶段定义必须具备的操作与状态。
- 原因：当前问题是“覆盖不齐”，不是“某个算法错误”，清单化可避免遗漏。
- 备选：直接按代码目录修 bug。
- 未选原因：容易遗漏跨模块行为，且难验证“无简化”。

### Decision 2: 统一状态源与派生展示，禁止各入口自定义语义
- 方案：以 `stores/workflow` 和 `stores/workflow-editor` 的统一状态语义为主，侧边栏、主页面、执行卡片、历史面板只消费同一套状态映射。
- 原因：减少“同一执行在不同入口状态不一致”。
- 备选：每个入口自行适配运行时事件。
- 未选原因：会继续产生语义漂移与隐藏分叉逻辑。

### Decision 3: 为关键断点定义强制错误反馈与恢复路径
- 方案：对保存失败、执行失败、调度创建失败、触发同步失败等路径统一要求：错误可见、原因可定位、可重试或回退。
- 原因：现有简化路径常表现为 silent failure，用户无法判断当前状态。
- 备选：仅记录日志，不暴露 UI 反馈。
- 未选原因：不满足“完整可用”目标。

### Decision 4: 测试策略采用“场景对齐规格”
- 方案：每个 specs 场景对应至少一个测试（Jest 或 E2E）；跨入口一致性与端到端闭环优先使用 E2E。
- 原因：需求与测试一一对应，便于后续回归审计。
- 备选：以模块级单测为主。
- 未选原因：无法证明多入口一致性和流程完整性。

## Risks / Trade-offs

- [风险] 状态语义收敛可能影响已有 UI 分支逻辑。
  - 缓解：增量迁移，先加兼容映射与回归测试，再删除旧分支。
- [风险] 补齐错误处理会暴露更多历史问题，短期告警变多。
  - 缓解：按错误类型分级展示，先保证可恢复性，再优化错误文案。
- [权衡] 增加 E2E 场景会拉长 CI 时间。
  - 缓解：分层测试，保留最关键闭环 E2E，其余用单测覆盖。

## Migration Plan

1. 建立完整性清单与状态映射表，标注每个环节“已实现/缺失/简化”。
2. 先补齐状态与错误反馈链路（低风险高收益），保证可观测。
3. 按生命周期顺序补齐功能：Authoring → Execution → Observability → Scheduling。
4. 同步补全测试：每补齐一个能力就增加对应验证用例。
5. 开启回归验证并观察一轮真实使用，再清理兼容分支。

## Open Questions

- 运行时事件模型中是否存在前端无法区分的失败类型，需要在 Tauri 层新增错误码映射？
- 调度与触发同步是否需要“最终一致性状态”提示（例如 pending/syncing）来降低误判？
- 是否需要为“完整性清单”增加开发期检查脚本，避免新功能再次出现简化路径？
